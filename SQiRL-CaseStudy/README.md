# SQiRL Case Study

Extension of the SQiRL RL environment for SQL injection detection, demonstrating evaluation pitfalls in DRL-based web security testing from the paper: **"SoK: The Pitfalls of Deep Reinforcement Learning for Cybersecurity"**

## Description

SQiRL is a grey-box SQL injection vulnerability scanner that uses deep reinforcement learning (DQN with Random Network Distillation) to intelligently fuzz input fields on web applications. Multiple worker agents generate targeted SQL injection payloads, guided by feedback from the application's database logs.

This case study (Section 7.4 of the paper) demonstrates the Environment Complexity pitfall (E-EC): how training and evaluating in overly simplified environments can inflate performance metrics that do not translate to real-world capabilities. Specifically, we compare agents trained with and without experience of sanitized (defended) SQL injection endpoints, then evaluate both on the full SQLi MicroBenchmark (SMB) containing both sanitized and non-sanitized vulnerabilities.

Two training conditions are used across 20 agents each:

- **Combined** (paper) / `training.php` (code): The default SMB training page, which includes both sanitized endpoints (`sqli14.php`-`sqli20.php`) and non-sanitized endpoints (`sqli1.php`-`sqli13.php`, `sqli21.php`-`sqli29.php`)
- **Non-Sanitized** (paper) / `training_no_san.php` (code): A modified training page where the sanitized endpoints are replaced with non-sanitized variants (`sqli14_no_san.php`-`sqli20_no_san.php`), so the agent never encounters sanitization defenses during training

> **Note on naming:** The `train_agents.sh` script and output directories use the label "sanitized" for what the paper calls **"Combined"** (because `training.php` *includes* sanitized endpoints), and "non_san" for what the paper calls **"Non-Sanitized"**. The directory `stats_logs_sanitized/` corresponds to the paper's *Combined* condition, and `stats_logs_non_san/` corresponds to the paper's *Non-Sanitized* condition.

Table 2 in the paper shows that while both agents discover vulnerabilities in simplified non-sanitized settings (~88%), performance drops sharply when evaluated on sanitized endpoints: a 47.4% drop for the *Combined* agent and a further 11.3% for the *Non-Sanitized* agent. This demonstrates that performance can be over-inflated by evaluating in simplified environments, leading to promising results that do not translate into real-world capabilities.

The modifications made to the upstream SQiRL code are captured in `case_study.diff`.

## Requirements

### 1. Set up the SQLi MicroBenchmark

The SQLi MicroBenchmark is a Docker-based PHP/MySQL web application with known SQL injection vulnerabilities. Install [Docker](https://docs.docker.com/engine/install/), then:

```bash
cd Modified-SQiRL/SQLiMicrobenchmark
docker compose up -d
```

The benchmark will be accessible at [http://localhost:8000](http://localhost:8000). On first run, you may need to fix permissions on the MySQL general log:

```bash
chmod 777 SQLiMicrobenchmark/mysql/general.log
```

### 2. Install Python dependencies

SQiRL requires Python 3.8:

```bash
cd Modified-SQiRL
pip install -r requirements.txt
```

### 3. Enable MySQL general logging

The SQiRL agent reads the MySQL general query log to observe SQL statements generated by its payloads. This is handled automatically by the Docker container, but if running MySQL separately, enable it:

```sql
SET GLOBAL general_log = 'ON';
```

Find the log file location with:
```bash
mysql -se "SHOW VARIABLES" -u root -p | grep -e general_log
```

## Training

All commands should be run from the `Modified-SQiRL/` directory.

### Running the full training experiment

The `train_agents.sh` script automates the full case study experiment, training 20 DQN_RND agents on each of two training endpoints:

```bash
cd Modified-SQiRL
./train_agents.sh
```

This runs two phases:
1. **Phase 1, Combined** (paper) / "sanitized" (code): 20 agents trained on `training.php` (includes both sanitized and non-sanitized endpoints)
2. **Phase 2, Non-Sanitized**: 20 agents trained on `training_no_san.php` (sanitized endpoints replaced with non-sanitized variants)

Training results are organized into:
- `stats_logs_sanitized/` - logs from Combined training (paper's "Combined" condition)
- `stats_logs_non_san/` - logs from Non-Sanitized training (paper's "Non-Sanitized" condition)

Each agent produces:
- Model checkpoints in `DQN_RND_Agent_Checkpoint/`
- Training statistics (`result_stats_*.stats`)
- Action logs (`actions_Applied_*.stats`)
- SQL statements (`sql_statment_*.stats`)
- Reward logs (`rewards_*.stats`)

**Training configuration:**

| Parameter | Value |
|---|---|
| Agent type | DQN_RND (type 2) |
| Episodes | 200 |
| Max timesteps per episode | 30 |
| Win criteria | 14 |
| Loss criteria | 200 |

### Training a single agent manually

```bash
python3 sqirl.py -u http://localhost:8000/training.php -v 2 \
    --log_file ./SQLiMicrobenchmark/mysql/general.log \
    --loss_criteria 200 --win_criteria 14 --agent 2 \
    --learning True --episodes 200 --max_timestamp 30
```

## Testing

### Running the full evaluation

The `test_agents.sh` script evaluates all trained agents against the `no_feedback.php` endpoint, which links to all SQLi endpoints in the SMB without providing error messages or feedback to the agent:

```bash
cd Modified-SQiRL
./test_agents.sh
```

This loads each trained agent's checkpoint from both `stats_logs_non_san/` (paper's Non-Sanitized) and `stats_logs_sanitized/` (paper's Combined) directories and evaluates them on the blind SQLi endpoint. The agent crawls `no_feedback.php`, discovers all input forms, and tests each one. Results are saved to `test_results/` with metadata tracking the source training condition.

The paper's Table 2 splits results into "Sanitized" and "Non-Sanitized" **evaluation environments**. This split is produced during analysis (not during testing): `analyze_results.py` groups per-endpoint results by whether the endpoint includes sanitization (e.g., `sqli14.php`-`sqli18.php` are sanitized) or not.

**Evaluation configuration:**

| Parameter | Value |
|---|---|
| Test endpoint | `http://localhost:8000/no_feedback.php` |
| Episodes | 10 |
| Max timesteps per episode | 10 |
| Win criteria | 1 |
| Loss criteria | 10 |

### Testing a single agent manually

```bash
python3 sqirl.py -u http://localhost:8000/no_feedback.php -v 2 \
    --log_file ./SQLiMicrobenchmark/mysql/general.log \
    --model_dir ./stats_logs_sanitized/<run_dir>/DQN_RND_Agent_Checkpoint \
    --learning False --episodes 10 --max_timestamp 10 \
    --loss_criteria 10 --win_criteria 1
```

## Analysis

### Analyzing test results

`analyze_results.py` processes all test results and computes per-endpoint win rate and episode length with 95% confidence intervals, separated by training condition (sanitized vs non-sanitized):

```bash
cd Modified-SQiRL
python3 analyze_results.py ./test_results
```

This produces:
- Terminal output with color-coded comparison tables
- `results_analysis.csv` - full results as CSV
- `results_table.tex` - LaTeX table for paper inclusion
- `results_table_compact.tex` - compact LaTeX table

The analysis includes per-endpoint breakdowns, a sanitization subset summary (endpoints sqli14-18, sqli26-29), and aggregate totals across all endpoints. The "Sanitized" and "Non-Sanitized" columns in the paper's Table 2 correspond to the sanitization subset and remaining endpoints respectively in the analysis output.
